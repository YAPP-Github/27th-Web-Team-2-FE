# Feature Specification: 모바일 뷰포트 높이 개선

**Feature Branch**: `feat/#121-mobile-viewport-height`
**Created**: 2026-03-26
**Status**: Draft
**GitHub Issue**: #121
**Input**: 모바일 환경에 맞게 view height 개선 - dvh 등을 사용하여 모바일 브라우저의 동적 뷰포트 높이에 대응

## 배경

현재 서비스의 모든 페이지에서 `min-h-screen` (= `min-height: 100vh`)을 사용하고 있다. 모바일 브라우저(Chrome, Safari 등)에서는 주소창과 하단 네비게이션 바가 스크롤에 따라 동적으로 나타나고 사라진다. `100vh`는 이러한 동적 UI 요소를 고려하지 않는 고정된 뷰포트 높이를 반환하기 때문에, 하단에 배치된 CTA 버튼이나 콘텐츠가 브라우저 네비게이션 바에 의해 가려지는 문제가 발생한다.

### 영향받는 페이지

- 온보딩 페이지 (`/`) - "시작하기" 버튼이 잘림
- 모임 생성 페이지 - 하단 "다음" 버튼이 잘림
- 날짜 선택 페이지 - 하단 "완료" 버튼이 잘림
- 참여자 이름 입력 페이지 - 하단 버튼이 잘림
- 참여자 날짜 선택 페이지 - 하단 버튼이 잘림
- 참여자 편집 페이지들 - 하단 버튼이 잘림
- 투표 결과 페이지 (`/meet/[meetingId]`) - 하단 콘텐츠가 잘림

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 모바일에서 하단 버튼이 항상 보이도록 개선 (Priority: P1)

사용자가 모바일 브라우저에서 서비스에 접속했을 때, 주소창이나 하단 네비게이션 바의 표시 여부와 관계없이 페이지의 모든 콘텐츠(특히 하단 CTA 버튼)가 화면 내에 온전히 표시되어야 한다.

**Why this priority**: 하단 버튼이 잘리면 사용자가 핵심 행동(시작하기, 다음 단계 진행, 투표 완료)을 수행할 수 없어 서비스 이용 자체가 불가능해진다. 이는 가장 치명적인 UX 문제다.

**Independent Test**: 모바일 브라우저(Chrome, Safari)에서 각 페이지에 접속하여 주소창이 표시된 상태에서도 하단 버튼이 완전히 보이는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 모바일 Chrome에서 온보딩 페이지에 접속한 상태, **When** 주소창과 하단 네비게이션 바가 모두 표시된 상태, **Then** "시작하기" 버튼이 브라우저 UI에 가려지지 않고 완전히 보여야 한다.
2. **Given** 사용자가 모바일 Safari에서 모임 생성 페이지에 접속한 상태, **When** 하단 툴바가 표시된 상태, **Then** "다음" 버튼이 가려지지 않고 완전히 보여야 한다.
3. **Given** 사용자가 모바일 브라우저에서 콘텐츠를 스크롤하는 중, **When** 주소창이 사라지거나 다시 나타날 때, **Then** 페이지 레이아웃이 자연스럽게 조정되어 콘텐츠가 잘리거나 빈 공간이 생기지 않아야 한다.

---

### User Story 2 - 데스크톱 환경에서 기존 레이아웃 유지 (Priority: P2)

데스크톱 브라우저 사용자의 기존 경험이 변경되지 않아야 한다. 뷰포트 높이 개선은 모바일에서만 의미가 있으며, 데스크톱에서는 기존과 동일한 레이아웃을 유지해야 한다.

**Why this priority**: 모바일 개선을 위해 데스크톱 경험이 퇴보하면 안 된다. 회귀 방지가 중요하다.

**Independent Test**: 데스크톱 브라우저에서 모든 페이지를 순회하며 기존 레이아웃과 동일하게 표시되는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 사용자가 데스크톱 Chrome에서 서비스에 접속한 상태, **When** 각 페이지를 탐색할 때, **Then** 모든 페이지의 레이아웃이 변경 전과 동일하게 표시되어야 한다.

---

### Edge Cases

- 주소창이 나타나고 사라지는 과도 상태에서 레이아웃이 깨지지 않아야 한다.
- 화면 회전(가로/세로 전환) 시 뷰포트 높이가 올바르게 재계산되어야 한다.
- `dvh`를 지원하지 않는 구형 브라우저에서 기존 `vh`로 폴백되어 현재와 동일하게 동작해야 한다.
- PWA(홈 화면 추가) 모드에서도 정상 동작해야 한다.
- 키보드가 올라온 상태에서는 뷰포트 높이를 변경하지 않고 기존 높이를 유지해야 한다 (레이아웃 점프 방지).

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 모든 전체 높이 레이아웃 페이지에서 뷰포트 높이가 모바일 브라우저의 동적 UI 요소(주소창, 네비게이션 바)를 고려하여 계산되어야 한다.
- **FR-002**: 하단 CTA 버튼이 모바일 브라우저의 동적 UI 요소에 의해 가려지지 않아야 한다.
- **FR-003**: 동적 뷰포트 높이를 지원하지 않는 브라우저에서 기존 동작(100vh 기반)으로 자동 폴백되어야 한다.
- **FR-004**: 데스크톱 브라우저에서 기존 레이아웃이 변경되지 않아야 한다.
- **FR-005**: 모바일 브라우저에서 주소창이 나타나거나 사라질 때 레이아웃이 부드럽게 전환되어야 한다.
- **FR-006**: 키보드가 올라올 때 뷰포트 높이가 변경되지 않아야 한다. 키보드 표시 시 레이아웃 점프가 발생하지 않도록 기존 높이를 유지한다.

### Assumptions

- 대상 브라우저: iOS Safari 15.4+, Android Chrome 108+ (dvh 지원 브라우저)
- 서비스가 모바일 우선(mobile-first) 서비스이므로 모바일 환경의 UX가 최우선이다.
- 현재 `min-h-screen`을 사용하는 모든 페이지가 개선 대상이다.

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 모바일 브라우저(Chrome, Safari)에서 모든 페이지의 하단 CTA 버튼이 브라우저 UI에 가려지지 않고 100% 보여야 한다.
- **SC-002**: 뷰포트 높이 변경 시(주소창 표시/숨김) 레이아웃 전환이 시각적으로 끊김 없이 이루어져야 한다.
- **SC-003**: 데스크톱 브라우저에서 변경 전후 레이아웃에 시각적 차이가 없어야 한다.
- **SC-004**: dvh 미지원 브라우저에서 기존과 동일하게 동작하여 사용자 경험이 퇴보하지 않아야 한다.
